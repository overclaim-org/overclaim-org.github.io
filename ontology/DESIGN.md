# Design: Epistemic Overclaim Ontology

The durable design rationale for this repo: what it is, why it is shaped this way,
and the rules that govern it. Operational guidance for working in the repo lives in
`AGENTS.md`; the academic prior-art map and citation registry live in
`PRIOR_ART_LITERATURE.md`.

## What this is

A versioned **ontology of epistemic-overclaim failure modes**: named codes for the
ways confident language outruns its evidence, in LLM and human prose. The
deliverable is the taxonomy, not a runtime.

## Why the ontology is the asset, not an engine

The project originates from a Reddit thread on avoiding AI-induced delusion, which
links a single ~52KB Python gist (`PUBLIC LIGHTHOUSE CORE`) framed as "flight
envelope protection for AI output", a productized slice of WillowEmberly's
two-year Negentropy project (Negentropic Reasoning Protocol, Open Cognitive
Avionics Package). The `grounding.yaml` pack is distilled from that work, which is
shared openly and licenses OCAP as CC0.

The gist's value is its vocabulary, not its code. Both of its detection layers are
shallow:

1. Free-text detection is keyword bingo: a few hard-coded word-lists plus inline
   `contains_any` checks. The polished demo output that tags phrases like "leading
   researchers" is not producible by that code; it was generated by an LLM told to
   load the framework and evaluate.
2. The structured modules do not read text at all. They read pre-set booleans the
   caller supplies (`retroactive_rewrite: True`) and echo a code. Detection is
   `if flag:`.

So the real detector is the LLM, and the file is a rubric it reads. This repo makes
that explicit: **the YAML packs are the source of truth, the LLM is the detector,
and the Python is a renderer that compiles packs into a paste-in rubric and never
reads subject text.** Do not rebuild the home-grown engine. The repo is a versioned
ontology plus thin adapters that emit into tools that already exist.

## What is novel here

The individual codes mostly have established academic or legal homes (see
`PRIOR_ART_LITERATURE.md`); the contribution is the synthesis and delivery, not the
codes in isolation. Position it as:

> A unified, frame-aware operational rubric for detecting epistemic overclaim in
> prose, synthesizing established work on hedging, factuality, spin, hallucination,
> propaganda techniques, advertising substantiation, automation bias, and
> requirements quality into a practical code vocabulary an LLM or reviewer applies
> to a declared genre and goal.

Not as "a novel taxonomy of epistemic overclaim". The defensible, narrower pieces:

1. The cross-domain synthesis into one named code vocabulary.
2. The genre-relative **frame gate** (severity scored against the declared
   purpose).
3. The delivery model: ontology pasted into a context window, LLM applies it.
4. A few management/grounding formulations that combine known concepts not
   obviously named as standard taxonomy members.

## Delivery: overclaim.org as an agent-facing domain

The delivery model (point 3 above) is the reframe that names the project. The
ontology is consumed by an LLM, not read by a human: render a rubric, paste it plus
the subject text into a context window, the model applies it. That is the signature
of an *agent-facing domain*, a site whose primary consumer is a model rather than a
person: the homepage leads with a code block, the artifacts are markdown and YAML at
stable URLs, and an `llms.txt` sits next to them. overclaim.org is built as one, not
as a human docs site that happens to describe a taxonomy.

The published surface:

- **`/llms.txt`** , the anchor file (per the llms.txt convention): title, one-line
  summary, curated links to the `.md` rubric variants and the raw packs. The first
  thing an agent pulls.
- **`/rubric/<genre>.md`** , the rendered, paste-ready rubric per declared frame
  (`analytical`, `advocacy`, `sales`, `status`, `persuasion`). `render/rubric.py`
  stops being a local CLI and becomes a **build step**: packs in, static markdown
  out, regenerated on every site build so the served rubric never drifts from the
  packs.
- **`/packs/<pack>.yaml`** , the raw packs served verbatim as the machine-readable
  source of truth, for a caller that wants to compile its own rubric or wire the
  codes into DeepEval / Guardrails.
- **Human docs** (this DESIGN, the prior-art map, a browsable code catalog) , the
  *secondary* surface, same MkDocs-on-Pages site. People land here; agents land on
  `/llms.txt` and `/rubric/*`.

The mechanics are the standard home-project pattern (MkDocs Material on GitHub Pages,
Porkbun DNS): the whole site is static markdown and YAML, and the rubric build is one
generation step before `mkdocs build`. The one real "push-instructions" channel an
agent honours is MCP, so the natural v2 is an MCP endpoint (`mcp.overclaim.org`)
exposing the rubric as a tool or prompt; the static `llms.txt` + rubric URLs are v1
and need no server.

## Prior art (researched; do not redo)

The surrounding space splits into two mature camps nobody has fused this way.

- **Governance / eval runtimes** validate LLM output through modules and metrics:
  DeepEval, RAGAS (faithfulness = claims grounded in context), Guardrails,
  NeMo-Guardrails, TruLens (groundedness), llm-guard. These operationalize the
  confidence-vs-evidence axis as numeric scores, but none ship a named taxonomy of
  rhetorical overclaim modes.
- **Reasoning-failure taxonomies** name the failures academically: the LOGIC
  dataset's 13-class fallacy taxonomy, causal-reasoning evals, clinical-trial
  "spin" detection, hedge-cue detection (CoNLL-2010 / BioScope). These ship as
  datasets and classifiers, not a paste-in rubric.

The gap, and this repo's wedge: a named rhetorical-overclaim taxonomy sitting on
top of a real groundedness scorer. The RAGAS/TruLens bridge is the point: the
overclaim codes are the qualitative layer on top of their quantitative
groundedness score. The README should lead with that combination.

## The ontology

54 codes across 7 packs. The cross-cutting packs are `core.yaml`
(epistemic-overclaim core), `substantiation.yaml` (shared proof-bar axis), and
`grounding.yaml` (reasoning-environment "warning lights", CC0-attributed to
WillowEmberly / Negentropy). The per-domain packs are
`packs/{engineering,sales,marketing,management}.yaml`, each adding only modes the
cross-cutting packs do not already name; cross-domain reuse goes through a code's
`domains` tag, not by copying codes between packs.

Avionics and agentic-execution codes from the source corpus (`ORIENTATION_*`,
`EXECUTION_*`, `STRUCTURE_*`, `STATE_*`, process-risk, system codes) are
deliberately excluded: they are about agent execution and spatial control, not
prose overclaim. They can return later as a separate optional "agentic-execution"
pack if there is demand.

`MANAGEMENT_COMMITMENT_WITHOUT_MECHANISM` is a headline addition: a promise verb
with no owner, date, or first step ("I'll make sure", "put a plan around it"). It
is distinct from `NARRATIVE_CONFIDENCE_EVIDENCE_DECOUPLED` (confidence outrunning
evidence) because the failure is *action* outrunning mechanism: the claim is a
commitment, and what is missing is the owner/date/first-step, not the evidence. It
was found by running the ontology over a real exec reply where five paragraphs of
intent verbs carried only two concrete commitments, and no existing code named that
pattern precisely.

## Frame gate (the central design rule)

The ontology judges text relative to its declared purpose, or it produces
true-but-misapplied findings, the classic reason critique tools get ignored.

1. **The frame gate runs first and modulates severity.** The caller declares
   `{genre, audience, goal}`. Each code carries `applies_in_frames`, giving its
   disposition per genre: `defect` (full severity), `advisory` (downgrade one notch,
   flag only when egregious), or `suppressed` (drop as expected for the genre). An
   overclaim in an analytical memo is a defect; the same line in advocacy/sales copy
   is often legitimate framing. Skipping this gate is itself a
   `PREFLIGHT_PURPOSE_MISSING` / `INPUT_FRAME_NOT_DECLARED` violation by the analyst.
2. **A declared limitation is a mitigation, not a violation.** An explicit,
   in-artifact hedge that caps confidence to match the evidence resolves the
   matching evidence/risk code via the `LIMITATION_DECLARED` state. It only counts
   if visible to the reader in the artifact, not in the author's later defence of
   it. Verify presence before clearing.

Both rules come from live tests. In one, an author had deliberately declared his
cost assumptions as weak (Rule 2 in action) and accepted a set of proof metrics as
the evidence layer to harden them later; the prior analysis had flagged the weak
assumptions as a defect, correct on facts but misweighted because it ignored the
advocacy frame and treated an intentional hedge as a failure. The tool had
under-applied its own preflight gate.

**Invariant:** substantiation codes and `COMMUNICATION_IDENTITY_BINDING` do not
soften in sales/advocacy. Persuasive claims carry the highest substantiation duty
(FTC "competent and reliable evidence" standard), and identity binding is the
anti-sycophancy guardrail. Their disposition stays `defect` in every genre.

## Contribution rule (prevents bloat)

A new code earns its place only if it is **true + unnamed + generalises**: a real
example trips a genuine pattern, no existing code names it precisely, and it holds
beyond that one case. Grow the ontology by running it over diverse genres (one
document per genre) and watching for the "true pattern, no precise code" signal,
not by bulk-collecting AI text, which re-confirms existing codes and biases toward
LLM stylistic tells. Examples serve calibration (gold pairs per code) and
gap-finding, not discovery-by-volume. The ontology is not AI-specific: it names
human rhetorical failures that AI reproduces, so keep the example net across AI and
human text.

## Proposed repo layout

```
overclaim/                     # published as overclaim.org
  ontology/
    core.yaml                  # epistemic-overclaim core
    substantiation.yaml        # shared substantiation axis
    grounding.yaml             # reasoning-environment warning lights (CC0-attributed)
    packs/engineering.yaml
    packs/sales.yaml
    packs/marketing.yaml
    packs/management.yaml
  schema/code.schema.json      # machine-readable code definition contract (planned)
  render/
    rubric.py                  # packs -> rubric markdown; the site build step (built)
    deepeval_metrics.py        # ontology -> DeepEval custom metrics (planned)
    guardrails_validators.py   # ontology -> Guardrails validators (planned)
  site/                        # MkDocs Material source for overclaim.org (planned)
    mkdocs.yml
    docs/
      index.md                 # human landing page
      llms.txt                 # agent anchor file (links rubric/*.md + packs/*.yaml)
      rubric/<genre>.md        # generated by render/rubric.py at build time
      packs/<pack>.yaml        # raw packs, served verbatim
  examples/                    # positive/negative example per code, eval fixtures (planned)
```

The draft currently keeps the packs at the repo root rather than under `ontology/`;
`render/rubric.py` resolves either layout. Move all packs together if introducing
the `ontology/` directory.

## Open questions

1. **Attribution / licensing , resolved.** Dual-license: the ontology content
   (`*.yaml`, the rubric markdown, docs) under **CC0 1.0** (`CC0-1.0`); the code
   (`render/rubric.py`, adapters) under **MIT**. CC0 matches the agent-facing goal
   (frictionless ingestion) and the upstream OCAP, which is itself CC0, so the licence
   chain stays clean; a software licence on the data would be a category error.
   Ship `LICENSE` (MIT) + `LICENSE-DATA` (CC0) and state the split in the README.
   Credit "WillowEmberly / Negentropy" in a NOTICE + the `grounding.yaml`
   `attribution` field (a courtesy, not a CC0 obligation). Remaining before publish:
   a courtesy heads-up to WillowEmberly, and confirm whether the gist owner is the
   same author or a permitted re-poster.
2. **Name , resolved.** The project is **overclaim**, published at **overclaim.org**
   (registered through the home-project flow). Drops the earlier `overclaim-ontology`
   / `claimguard` / `lighthouse-ontology` candidates: "overclaim" is shorter,
   claimable, and matches the domain (and sidesteps the "Lighthouse" collision with
   Google Lighthouse).
3. **Scope of v1 , resolved.** Ship **core + substantiation + grounding +
   management** (39 codes); defer **engineering + sales + marketing** (15) to v1.x. The
   discriminator is validation maturity, not domain coverage: the four shipped packs
   are battle-tested (core/substantiation are the validated spine; grounding cleared
   the contribution rule and carries the WillowEmberly attribution; management is
   validated against two live exec messages). The three deferred packs were added by
   reasoning rather than by running over diverse real genres (~35% / ~15% / ~15%
   completeness), so they have not cleared "true + unnamed + generalises" through real
   use. The rubric build step makes adding a pack later cheap and non-breaking, so v1
   needs to be correct, not complete.
4. **Agentic-execution carryover , resolved: dropped.** Out of scope. The avionics /
   agent-execution codes detect runtime behaviour (scope, permission, replayability,
   state invariants) from execution traces and state, not from prose, so they break
   the "LLM applies a rubric to subject text" delivery model. If ever pursued it is a
   separate project with its own execution-trace substrate, not a pack here.
5. **GitHub home , resolved.** A dedicated public org `overclaim` with repo
   `overclaim.github.io` (mirrors `openconsumables`). The org must be created by hand
   first (no API for org creation); the home-project flow guides that one step.

## Next steps

The ontology is usable end to end today: render a rubric, paste it plus the subject
text into a context window. Name is resolved (overclaim / overclaim.org). Remaining
work, roughly in order: resolve licensing; stand up the overclaim.org site (MkDocs on
Pages, plus the rubric build step that emits `/rubric/*.md` and `/llms.txt`);
battle-test the rendered rubric on real text in a fresh context (prompt quality is
unproven); then the DeepEval / Guardrails adapters (the quantitative-groundedness
bridge), a machine-readable code schema, and the `mcp.overclaim.org` endpoint.
Validate any new code against the contribution rule.
